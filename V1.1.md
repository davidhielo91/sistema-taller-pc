# V1.1 — Especificación de nuevas funcionalidades

> Complemento de `AGENTS.md`. Mantiene las mismas reglas: single-tenant (un
> despliegue por taller), stack sin cambios (Next.js 16 App Router + TypeScript +
> Prisma + PostgreSQL), configuración por entorno, interfaz en español, nada de
> multi-tenancy. **Antes de codificar, leer `AGENTS.md`, `CLAUDE.md` y `CHANGELOG.md`.**
> Las migraciones de esta versión son **aditivas y compatibles** con datos existentes.

## Alcance de la v1.1 (Nivel 1)

Cinco funcionalidades, de alto uso en el mostrador y sin cambiar el perfil de
despliegue (no se agrega manejo de archivos ni servicios externos):

1. **Bitácora de estados** — historial de cada cambio de estado (qué, cuándo, quién).
2. **Pagos y abonos** — registro de cobros parciales/totales por orden.
3. **Aviso por WhatsApp** — botón que abre WhatsApp con un mensaje prellenado (`wa.me`).
4. **Etiqueta imprimible** — etiqueta pequeña para pegar en el equipo en el estante.
5. **Reportes básicos** — ingresos, número de reparaciones, tiempo promedio y por técnico.

**Sigue fuera de alcance:** inventario completo de refacciones, facturación fiscal,
fotos del equipo, consulta pública por folio (se evaluarán en v1.2).

## Cambios al modelo de datos (Prisma)

Adiciones al `schema.prisma`. No se modifican ni eliminan campos existentes.

```prisma
enum EstadoPago {
  PENDIENTE
  ABONADO
  PAGADO
}

// --- Adiciones al model Orden existente ---
// estadoPago EstadoPago @default(PENDIENTE)
// historial  HistorialEstado[]
// pagos      Pago[]

// --- Adiciones al model Usuario existente ---
// historial HistorialEstado[]
// pagos     Pago[]

// --- Adiciones al model Ajustes existente (todos opcionales) ---
// codigoPaisWhatsapp String?  // ej. "52" (MX), "54" (AR), "57" (CO). Se antepone
//                             // al teléfono del cliente si no trae código de país.
// mensajeWhatsappListo String? // plantilla opcional del mensaje de "equipo listo".

model HistorialEstado {
  id             String       @id @default(cuid())
  orden          Orden        @relation(fields: [ordenId], references: [id])
  ordenId        String
  estadoAnterior EstadoOrden?
  estadoNuevo    EstadoOrden
  usuario        Usuario      @relation(fields: [usuarioId], references: [id])
  usuarioId      String
  createdAt      DateTime     @default(now())
}

model Pago {
  id        String   @id @default(cuid())
  orden     Orden    @relation(fields: [ordenId], references: [id])
  ordenId   String
  monto     Decimal  @db.Decimal(12, 2)
  metodo    String?  // campo libre: efectivo, transferencia, tarjeta, etc.
  nota      String?
  usuario   Usuario  @relation(fields: [usuarioId], references: [id])
  usuarioId String
  createdAt DateTime @default(now())
}
```

Notas de migración:
- Las órdenes existentes quedan con `estadoPago = PENDIENTE`. Correcto, no hay backfill.
- La bitácora arranca desde ahora; las órdenes previas no tendrán historial retroactivo.

## 1. Bitácora de estados

- Cada vez que se cambia el estado de una orden (acción `cambiarEstado` en
  `lib/actions/ordenes.ts`), además de actualizar `Orden.estado`, se crea una fila en
  `HistorialEstado` con `estadoAnterior`, `estadoNuevo` y el usuario en sesión.
  Hacerlo en una **transacción** para no dejar el estado y la bitácora desincronizados.
- En el detalle de la orden, mostrar una línea de tiempo (timeline) con los cambios:
  estado, fecha/hora (`formatDate`) y técnico, del más reciente al más antiguo.
- Esta bitácora es la base de los reportes de tiempo (sección 5).

## 2. Pagos y abonos

- Desde el detalle de la orden: formulario para **registrar un abono** (monto, método
  opcional, nota opcional). Cada abono crea una fila en `Pago` con el usuario en sesión.
- Mostrar la lista de pagos de la orden, el **total abonado** y el **saldo**
  (`costo − suma de pagos`), todo formateado con `formatCurrency`.
- Recalcular `Orden.estadoPago` cada vez que cambian los pagos o el `costo`:
  - sin pagos → `PENDIENTE`
  - pagos > 0 pero menores al costo → `ABONADO`
  - pagos ≥ costo → `PAGADO`
  - si `costo` es nulo, dejar en `PENDIENTE`.
- En el **listado de órdenes**: badge de estado de pago y filtro por estado de pago.
- Validar en servidor: monto > 0; no permitir registrar pagos en órdenes canceladas.

## 3. Aviso por WhatsApp

- Botón en el detalle de la orden: **"Avisar por WhatsApp"**, visible solo si el cliente
  tiene teléfono.
- Construye un enlace `https://wa.me/<numero>?text=<mensaje>`:
  - `numero`: teléfono del cliente, limpiado de espacios/guiones/paréntesis. Si no
    empieza con `+` ni con código de país, anteponer `Ajustes.codigoPaisWhatsapp`.
  - `mensaje`: usar `Ajustes.mensajeWhatsappListo` si existe; si no, un texto por
    defecto del tipo "Hola {cliente}, le informamos que su {tipoEquipo} (folio #{folio})
    ya está listo para recoger. Gracias." Sustituir las variables disponibles y
    codificar con `encodeURIComponent`.
- Sin API ni dependencias: es solo un enlace que abre WhatsApp (`target="_blank"`).
- Añadir `codigoPaisWhatsapp` y `mensajeWhatsappListo` a la pantalla de Ajustes (ADMIN).

## 4. Etiqueta imprimible

- Nueva ruta de impresión independiente (fuera del layout del dashboard), al estilo del
  comprobante: `app/etiqueta/[id]/page.tsx`, con verificación de sesión.
- Contenido mínimo y compacto, pensado para una etiqueta pequeña: **folio grande**,
  nombre del cliente, tipo de equipo (marca/modelo si hay) y fecha de ingreso.
  Opcional: el nombre del taller.
- Botón de imprimir como componente cliente (mismo patrón que `PrintButton`).
- Enlace "Etiqueta" en el detalle de la orden, abre en pestaña nueva.

## 5. Reportes básicos

- Nueva página `app/dashboard/reportes/page.tsx` con filtro de **rango de fechas**
  (por defecto, el mes actual).
- Métricas, calculadas con datos ya existentes + bitácora + pagos:
  - **Ingresos del periodo**: suma de `Pago.monto` con `createdAt` en el rango.
  - **Reparaciones**: número de órdenes con estado `ENTREGADO` y `fechaEntrega` en rango.
  - **Tiempo promedio de entrega**: promedio de (`fechaEntrega − fechaIngreso`) de las
    entregadas en el rango, expresado en días.
  - **Por técnico**: desglose de reparaciones entregadas por técnico (`entregadoPor`).
  - (Opcional, si la bitácora lo permite con claridad) tiempo promedio en cada estado.
- Solo lectura. No agrega dependencias de gráficas; basta con tablas y números. Si se
  quiere una gráfica simple, usar algo ligero, pero no es requisito.

  sola vez por página (no volver a cachear en variable de módulo).
- Cada cambio de estado y cada pago se escriben en **transacción** junto con sus efectos.
- Crear las migraciones con Prisma (no editar la base a mano) y verificar que el arranque
  turnkey (`migrate deploy` + `seed`) sigue funcionando.
- Respetar roles: registrar abonos y avisar por WhatsApp lo puede hacer cualquier técnico;
  editar Ajustes sigue siendo solo ADMIN.
- Actualizar `CHANGELOG.md` y, si corresponde, una nota en el `README.md` sobre las
  nuevas variables/opciones de Ajustes (código de país de WhatsApp).
